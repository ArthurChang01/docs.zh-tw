# <a name="wcf-bindings-and-transports"></a><span data-ttu-id="ca251-103">WCF 系結和傳輸</span><span class="sxs-lookup"><span data-stu-id="ca251-103">WCF bindings and transports</span></span>

[!INCLUDE [book-preview](../../../includes/book-preview.md)]

<span data-ttu-id="ca251-104">WCF 有許多不同的內建系結 *，這些系*結會指定不同的網路通訊協定、電傳格式和其他的執行詳細資料。</span><span class="sxs-lookup"><span data-stu-id="ca251-104">WCF has lots of different built-in *bindings* that specify different network protocols, wire formats, and other implementation details.</span></span> <span data-ttu-id="ca251-105">gRPC 實際上只有一個網路通訊協定和一個電傳格式（技術上來說，*可以*自訂電傳格式，但這不在本書的討論範圍內）。</span><span class="sxs-lookup"><span data-stu-id="ca251-105">gRPC effectively has just one network protocol and one wire format (technically the wire format *may* be customized, but that is beyond the scope of this book).</span></span> <span data-ttu-id="ca251-106">在大部分情況下，您可能會發現 gRPC 提供最佳的解決方案。</span><span class="sxs-lookup"><span data-stu-id="ca251-106">You are likely to discover that gRPC offers the best solution in most cases.</span></span> <span data-ttu-id="ca251-107">接下來是關於最相關的 WCF 系結，以及它們如何與 gRPC 中的對等項進行比較的簡短討論。</span><span class="sxs-lookup"><span data-stu-id="ca251-107">What follows is a short discussion about the most relevant WCF bindings and how they compare to their equivalent in gRPC.</span></span>

## <a name="nettcp"></a><span data-ttu-id="ca251-108">NetTCP</span><span class="sxs-lookup"><span data-stu-id="ca251-108">NetTCP</span></span>

<span data-ttu-id="ca251-109">WCF 的 NetTCP 系結允許持續連接、小型訊息和雙向訊息，但只能在 .NET 用戶端和伺服器之間運作。</span><span class="sxs-lookup"><span data-stu-id="ca251-109">WCF's NetTCP binding allows for persistent connections, small messages, and two-way messaging but only works between .NET clients and servers.</span></span> <span data-ttu-id="ca251-110">gRPC 允許相同的功能，但在多個程式設計語言和平臺上都支援。</span><span class="sxs-lookup"><span data-stu-id="ca251-110">gRPC allows the same functionality but is supported across multiple programming languages and platforms.</span></span> <span data-ttu-id="ca251-111">gRPC 有許多 WCF NetTCP 系結的功能，但不一定會以相同的方式來執行。</span><span class="sxs-lookup"><span data-stu-id="ca251-111">gRPC has many of the features of WCF NetTCP binding, although not always implemented in the same way.</span></span> <span data-ttu-id="ca251-112">例如，在 WCF 中，加密是透過設定來控制，並在架構中處理。</span><span class="sxs-lookup"><span data-stu-id="ca251-112">For example, in WCF, encryption is controlled through configuration and handled in the framework.</span></span> <span data-ttu-id="ca251-113">在 gRPC 中，加密是在透過 TLS 使用 HTTP/2 的連線層級達成。</span><span class="sxs-lookup"><span data-stu-id="ca251-113">In gRPC, encryption is achieved at the connection level using HTTP/2 over TLS.</span></span>

## <a name="http"></a><span data-ttu-id="ca251-114">HTTP</span><span class="sxs-lookup"><span data-stu-id="ca251-114">HTTP</span></span>

<span data-ttu-id="ca251-115">WCF 的 BasicHttpBinding 通常是以文字為基礎，使用 SOAP 做為電傳格式，而且非常慢于新式網路應用程式的標準。</span><span class="sxs-lookup"><span data-stu-id="ca251-115">WCF's BasicHttpBinding is generally text based, using SOAP as the wire format, and is very slow by the standards of modern networked applications.</span></span> <span data-ttu-id="ca251-116">它只是用來提供跨平臺互通性，或透過網際網路基礎結構的連線。</span><span class="sxs-lookup"><span data-stu-id="ca251-116">It's only really used to provide cross-platform interoperability, or connection over internet infrastructure.</span></span> <span data-ttu-id="ca251-117">GRPC 中的對等專案，因為它使用 HTTP/2 做為適用于訊息之 binary Protobuf 電傳格式的基礎傳輸層—可提供 NetTCP 服務層級效能，但具備所有新式程式設計語言的完整跨平臺互通性和集成.</span><span class="sxs-lookup"><span data-stu-id="ca251-117">The equivalent in gRPC—because it uses HTTP/2 as the underlying transport layer with the binary Protobuf wire format for messages—can offer NetTCP service level performance but with full cross-platform interoperability with all modern programming languages and frameworks.</span></span>

## <a name="named-pipes"></a><span data-ttu-id="ca251-118">具名管道</span><span class="sxs-lookup"><span data-stu-id="ca251-118">Named Pipes</span></span>

<span data-ttu-id="ca251-119">WCF 提供了具名管道系結，以便在相同實體電腦上的進程之間進行通訊。</span><span class="sxs-lookup"><span data-stu-id="ca251-119">WCF provided a Named Pipes binding for communication between processes on the same physical machine.</span></span> <span data-ttu-id="ca251-120">ASP.NET Core gRPC 的第一版不支援具名管道。</span><span class="sxs-lookup"><span data-stu-id="ca251-120">Named Pipes are not supported by the first release of ASP.NET Core gRPC.</span></span>

<span data-ttu-id="ca251-121">在 Windows 之外，具名管道提供的功能，通常是由 Unix 網域通訊端提供。</span><span class="sxs-lookup"><span data-stu-id="ca251-121">Outside Windows, the functionality provided by Named Pipes is instead generally provided by Unix Domain Sockets.</span></span> <span data-ttu-id="ca251-122">這些通訊端是一般 TCP 類似的通訊端，以檔案系統位址表示， `/var/run/docker.sock`例如，gRPC 可以同時搭配用戶端和伺服器使用。</span><span class="sxs-lookup"><span data-stu-id="ca251-122">These sockets are regular TCP-like sockets represented with file-system addresses, such as `/var/run/docker.sock`, which gRPC can work with as both client and server.</span></span> <span data-ttu-id="ca251-123">如果您需要在 Windows 上使用具名管道樣式功能，windows 10 和 Windows Server 的下一次更新 2019 Q4，會在 Windows 中新增網域通訊端做為完全支援的原生功能。</span><span class="sxs-lookup"><span data-stu-id="ca251-123">If you need to use Named Pipes style functionality on Windows, the next update to Windows 10 and Windows Server, in 2019 Q4, adds Domain Sockets as a fully supported native feature within Windows.</span></span> <span data-ttu-id="ca251-124">因此，在這些和更新版本的 Windows （或 Linux 上）上執行的 gRPC 服務可以使用網域通訊端，而不是具名管道。</span><span class="sxs-lookup"><span data-stu-id="ca251-124">So, gRPC services running on these and later versions of Windows (or on Linux) can use Domain Sockets instead of Named Pipes.</span></span> <span data-ttu-id="ca251-125">不過，如果您的小組無法更新至最新版本的 Windows，則您必須使用 localhost TCP 通訊端。</span><span class="sxs-lookup"><span data-stu-id="ca251-125">However, if your team is unable to update to the latest version of Windows, then you'll need to use localhost TCP sockets.</span></span> <span data-ttu-id="ca251-126">使用本機 TCP 通訊端的安全性疑慮，可以透過在用戶端和伺服器之間使用憑證驗證來解決。</span><span class="sxs-lookup"><span data-stu-id="ca251-126">Security concerns about using local TCP sockets can be addressed with the use of certificate authentication between client and server.</span></span>

## <a name="msmq"></a><span data-ttu-id="ca251-127">MSMQ</span><span class="sxs-lookup"><span data-stu-id="ca251-127">MSMQ</span></span>

<span data-ttu-id="ca251-128">MSMQ 是專屬的 Windows 訊息佇列。</span><span class="sxs-lookup"><span data-stu-id="ca251-128">MSMQ is a proprietary Windows message queue.</span></span> <span data-ttu-id="ca251-129">WCF 對 MSMQ 的系結可讓您從用戶端「引發並忘記」要求，未來可能會進行處理。</span><span class="sxs-lookup"><span data-stu-id="ca251-129">WCF's binding to MSMQ enables "fire and forget" requests from clients that may be processed at any time in the future.</span></span> <span data-ttu-id="ca251-130">gRPC 原本不提供任何訊息佇列功能。</span><span class="sxs-lookup"><span data-stu-id="ca251-130">gRPC doesn't natively provide any message queue functionality.</span></span> <span data-ttu-id="ca251-131">最佳的替代方法是直接使用訊息系統，例如 Azure 服務匯流排、RabbitMQ 或 Kafka。</span><span class="sxs-lookup"><span data-stu-id="ca251-131">The best alternative would be to directly use a messaging system like Azure Service Bus, RabbitMQ, or Kafka.</span></span> <span data-ttu-id="ca251-132">這可以透過用戶端直接將訊息放在佇列上，或 gRPC 用戶端串流服務來將訊息來執行。</span><span class="sxs-lookup"><span data-stu-id="ca251-132">This could be implemented with the client placing messages directly onto the queue, or a gRPC client streaming service that enqueues the messages.</span></span>

## <a name="webhttpbinding"></a><span data-ttu-id="ca251-133">WebHttpBinding</span><span class="sxs-lookup"><span data-stu-id="ca251-133">WebHttpBinding</span></span>

<span data-ttu-id="ca251-134">WebHttpBinding （也稱為「WCF ReST」）具有`WebGet`和屬性，可讓您開發 ReSTful api，這在這種情況下並`WebInvoke`不常見，因此可能會一次說出 JSON。</span><span class="sxs-lookup"><span data-stu-id="ca251-134">The WebHttpBinding (also known as WCF ReST), with the `WebGet` and `WebInvoke` attributes, enabled you to develop ReSTful APIs that could speak JSON at a time when this was less common than now.</span></span> <span data-ttu-id="ca251-135">因此，如果您有以 WCF REST 建立的 RESTful API，請考慮將它遷移至一般 ASP.NET Core MVC Web API 應用程式，這會提供對等的功能，而不是轉換成 gRPC。</span><span class="sxs-lookup"><span data-stu-id="ca251-135">Therefore, if you have a RESTful API built with WCF REST, consider migrating it to a regular ASP.NET Core MVC Web API application, which would provide equivalent functionality, rather than converting to gRPC.</span></span>
