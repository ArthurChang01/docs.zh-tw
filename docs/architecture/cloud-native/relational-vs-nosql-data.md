## <a name="the-cap-theorem"></a><span data-ttu-id="eb064-134">CAP 定理</span><span class="sxs-lookup"><span data-stu-id="eb064-134">The CAP theorem</span></span>

<span data-ttu-id="eb064-135">為了瞭解這些類型的資料庫之間的差異,請考慮 CAP 定理,這是一組應用於存儲狀態的分散式系統的原則。</span><span class="sxs-lookup"><span data-stu-id="eb064-135">As a way to understand the differences between these types of databases, consider the CAP theorem, a set of principles applied to distributed systems that store state.</span></span> <span data-ttu-id="eb064-136">圖 5-10 顯示了 CAP 定理的三個屬性。</span><span class="sxs-lookup"><span data-stu-id="eb064-136">Figure 5-10 shows the three properties of the CAP theorem.</span></span>

<span data-ttu-id="eb064-140">該定理指出,分散式數據系統將在一致性、可用性和分區容差之間進行權衡。</span><span class="sxs-lookup"><span data-stu-id="eb064-140">The theorem states that distributed data systems will offer a trade-off between consistency, availability, and partition tolerance.</span></span> <span data-ttu-id="eb064-141">而且,任何資料庫只能保證三個屬性中的*兩*個:</span><span class="sxs-lookup"><span data-stu-id="eb064-141">And, that any database can only guarantee *two* of the three properties:</span></span>

<span data-ttu-id="eb064-151">關係資料庫通常提供一致性和可用性,但不提供分區容差。</span><span class="sxs-lookup"><span data-stu-id="eb064-151">Relational databases typically provide consistency and availability, but not partition tolerance.</span></span> <span data-ttu-id="eb064-152">它們通常預配到單個伺服器,並通過向計算機添加更多資源來垂直擴展。</span><span class="sxs-lookup"><span data-stu-id="eb064-152">They're typically provisioned to a single server and scale vertically by adding more resources to the machine.</span></span>

<span data-ttu-id="eb064-153">許多關係資料庫系統都支援內建複製功能,其中主資料庫的副本可以製作到其他輔助伺服器實例。</span><span class="sxs-lookup"><span data-stu-id="eb064-153">Many relational database systems support built-in replication features where copies of the primary database can be made to other secondary server instances.</span></span> <span data-ttu-id="eb064-154">寫入操作對主實例執行,並複製到每個輔助實例。</span><span class="sxs-lookup"><span data-stu-id="eb064-154">Write operations are made to the primary instance and replicated to each of the secondaries.</span></span> <span data-ttu-id="eb064-155">發生故障時,主實例可能會故障轉移到輔助實例以提供高可用性。</span><span class="sxs-lookup"><span data-stu-id="eb064-155">Upon a failure, the primary instance can fail over to a secondary to provide high availability.</span></span> <span data-ttu-id="eb064-156">輔助資料庫還可用於分發讀取操作。</span><span class="sxs-lookup"><span data-stu-id="eb064-156">Secondaries can also be used to distribute read operations.</span></span> <span data-ttu-id="eb064-157">雖然寫入操作始終針對主副本,但讀取操作可以路由到任何輔助副本,以減少系統負載。</span><span class="sxs-lookup"><span data-stu-id="eb064-157">While writes operations always go against the primary replica, read operations can be routed to any of the secondaries to reduce system load.</span></span>

<span data-ttu-id="eb064-158">資料也可以跨多個節點水準分割區,例如分[片](https://docs.microsoft.com/azure/sql-database/sql-database-elastic-scale-introduction)。</span><span class="sxs-lookup"><span data-stu-id="eb064-158">Data can also be horizontally partitioned across multiple nodes, such as with [sharding](https://docs.microsoft.com/azure/sql-database/sql-database-elastic-scale-introduction).</span></span> <span data-ttu-id="eb064-159">但是,分片化通過將數據吐出許多不易通信的片段,大大增加了操作開銷。</span><span class="sxs-lookup"><span data-stu-id="eb064-159">But, sharding dramatically increases operational overhead by spitting data across many pieces that cannot easily communicate.</span></span> <span data-ttu-id="eb064-160">管理可能既昂貴又耗時。</span><span class="sxs-lookup"><span data-stu-id="eb064-160">It can be costly and time consuming to manage.</span></span> <span data-ttu-id="eb064-161">它最終可能會影響性能、表聯接和參考完整性。</span><span class="sxs-lookup"><span data-stu-id="eb064-161">It can end up impacting performance, table joins, and referential integrity.</span></span>

<span data-ttu-id="eb064-162">如果資料複本在"高度一致"的關係資料庫群集中失去網路連接,您將無法寫入資料庫。</span><span class="sxs-lookup"><span data-stu-id="eb064-162">If data replicas were to lose network connectivity in a "highly consistent" relational database cluster, you wouldn't be able to write to the database.</span></span> <span data-ttu-id="eb064-163">系統將拒絕寫入操作,因為它無法將該更改複製到其他數據副本。</span><span class="sxs-lookup"><span data-stu-id="eb064-163">The system would reject the write operation as it can't replicate that change to the other data replica.</span></span> <span data-ttu-id="eb064-164">每個數據副本都必須在事務完成之前進行更新。</span><span class="sxs-lookup"><span data-stu-id="eb064-164">Every data replica has to update before the transaction can complete.</span></span>

<span data-ttu-id="eb064-165">NoSQL 資料庫通常支援高可用性和分區容差。</span><span class="sxs-lookup"><span data-stu-id="eb064-165">NoSQL databases typically support high availability and partition tolerance.</span></span> <span data-ttu-id="eb064-166">它們橫向擴展,通常跨商品伺服器擴展。</span><span class="sxs-lookup"><span data-stu-id="eb064-166">They scale out horizontally, often across commodity servers.</span></span> <span data-ttu-id="eb064-167">這種方法在地理區域內和跨地理區域之間提供巨大的可用性,成本降低。</span><span class="sxs-lookup"><span data-stu-id="eb064-167">This approach provides tremendous availability, both within and across geographical regions at a reduced cost.</span></span> <span data-ttu-id="eb064-168">跨這些計算機或節點對數據進行分區和複製,從而提供冗餘和容錯能力。</span><span class="sxs-lookup"><span data-stu-id="eb064-168">You partition and replicate data across these machines, or nodes, providing redundancy and fault tolerance.</span></span> <span data-ttu-id="eb064-169">缺點是一致性。</span><span class="sxs-lookup"><span data-stu-id="eb064-169">The downside is consistency.</span></span> <span data-ttu-id="eb064-170">更改一個 NoSQL 節點上的數據可能需要一些時間才能傳播到其他節點。</span><span class="sxs-lookup"><span data-stu-id="eb064-170">A change to data on one NoSQL node can take some time to propagate to other nodes.</span></span> <span data-ttu-id="eb064-171">通常,NoSQL 資料庫節點會立即回應查詢 - 即使顯示的數據已過時且尚未更新。</span><span class="sxs-lookup"><span data-stu-id="eb064-171">Typically, a NoSQL database node will provide an immediate response to a query - even if the data that is presented is stale and hasn't updated yet.</span></span>
