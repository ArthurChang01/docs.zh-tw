# <a name="data-sovereignty-per-microservice"></a><span data-ttu-id="95c19-105">每個微服務的資料自主性</span><span class="sxs-lookup"><span data-stu-id="95c19-105">Data sovereignty per microservice</span></span>

<span data-ttu-id="95c19-106">微服務架構的重要規則是，每個微服務必須擁有自己的網域資料和邏輯。</span><span class="sxs-lookup"><span data-stu-id="95c19-106">An important rule for microservices architecture is that each microservice must own its domain data and logic.</span></span> <span data-ttu-id="95c19-107">就像完整的應用程式擁有其邏輯和資料，每個微服務也必須在自發的生命週期裡擁有其邏輯和資料，並且每個微服務有獨立的部署。</span><span class="sxs-lookup"><span data-stu-id="95c19-107">Just as a full application owns its logic and data, so must each microservice own its logic and data under an autonomous lifecycle, with independent deployment per microservice.</span></span>

<span data-ttu-id="95c19-108">這表示，網域的概念模型將在各個子系統或微服務之間有所不同。</span><span class="sxs-lookup"><span data-stu-id="95c19-108">This means that the conceptual model of the domain will differ between subsystems or microservices.</span></span> <span data-ttu-id="95c19-109">請考慮企業應用程式，其中客戶關係管理 (CRM) 應用程式、交易式購買子系統和客戶支援子系統每個都呼叫不同的客戶實體屬性和資料，而每一者都使用不同的繫結內容 (BC)。</span><span class="sxs-lookup"><span data-stu-id="95c19-109">Consider enterprise applications, where customer relationship management (CRM) applications, transactional purchase subsystems, and customer support subsystems each call on unique customer entity attributes and data, and where each employs a different Bounded Context (BC).</span></span>

<span data-ttu-id="95c19-110">這個原則非常類似[網域導向設計 (DDD)](https://en.wikipedia.org/wiki/Domain-driven_design)，其中每個[繫結內容](https://martinfowler.com/bliki/BoundedContext.html)或自發子系統或服務必須擁有其網域模型 (資料加上邏輯和行為)。</span><span class="sxs-lookup"><span data-stu-id="95c19-110">This principle is similar in [Domain-driven design (DDD)](https://en.wikipedia.org/wiki/Domain-driven_design), where each [Bounded Context](https://martinfowler.com/bliki/BoundedContext.html) or autonomous subsystem or service must own its domain model (data plus logic and behavior).</span></span> <span data-ttu-id="95c19-111">每個 DDD 繫結內容相互關聯到一個商務微服務 (一或多個服務)。</span><span class="sxs-lookup"><span data-stu-id="95c19-111">Each DDD Bounded Context correlates to one business microservice (one or several services).</span></span> <span data-ttu-id="95c19-112">我們會在下一節從這點繼續展開繫結內容模式。</span><span class="sxs-lookup"><span data-stu-id="95c19-112">This point about the Bounded Context pattern is expanded in the next section.</span></span>

<span data-ttu-id="95c19-113">在另一方面，許多應用程式中使用的傳統 (單一資料) 方法，是單一的集中式資料庫或幾個資料庫。</span><span class="sxs-lookup"><span data-stu-id="95c19-113">On the other hand, the traditional (monolithic data) approach used in many applications is to have a single centralized database or just a few databases.</span></span> <span data-ttu-id="95c19-114">這通常是標準化的 SQL 資料庫，用於整個應用程式及其所有內部子系統，如圖 4-7 所示。</span><span class="sxs-lookup"><span data-stu-id="95c19-114">This is often a normalized SQL database that's used for the whole application and all its internal subsystems, as shown in Figure 4-7.</span></span>

![顯示兩個資料庫方法的圖表。](./media/data-sovereignty-per-microservice/data-sovereignty-comparison.png)

<span data-ttu-id="95c19-116">**圖 4-7**.</span><span class="sxs-lookup"><span data-stu-id="95c19-116">**Figure 4-7**.</span></span> <span data-ttu-id="95c19-117">資料自主性比較：單一資料庫與微服務</span><span class="sxs-lookup"><span data-stu-id="95c19-117">Data sovereignty comparison: monolithic database versus microservices</span></span>

<span data-ttu-id="95c19-118">在傳統的方法中，所有服務之間會共用單一資料庫，通常使用分層架構。</span><span class="sxs-lookup"><span data-stu-id="95c19-118">In the traditional approach, there's a single database shared across all services, typically in a tiered architecture.</span></span> <span data-ttu-id="95c19-119">在微服務方法中，每個微服務都會擁有其模型/資料。</span><span class="sxs-lookup"><span data-stu-id="95c19-119">In the microservices approach, each microservice owns its model/data.</span></span> <span data-ttu-id="95c19-120">集中式資料庫方法一開始看起來較簡單，而且似乎能重複使用不同子系統中的實體，讓一切一致。</span><span class="sxs-lookup"><span data-stu-id="95c19-120">The centralized database approach initially looks simpler and seems to enable reuse of entities in different subsystems to make everything consistent.</span></span> <span data-ttu-id="95c19-121">但事實上，您會得到很大的資料表，它們會服務許多不同子系統，且包含在大部分情況下不需要的屬性和資料行。</span><span class="sxs-lookup"><span data-stu-id="95c19-121">But the reality is you end up with huge tables that serve many different subsystems, and that include attributes and columns that aren't needed in most cases.</span></span> <span data-ttu-id="95c19-122">就像是嘗試使用相同的實體地圖來進行短距離的健走、一整天的汽車旅行，以及學習地理。</span><span class="sxs-lookup"><span data-stu-id="95c19-122">It's like trying to use the same physical map for hiking a short trail, taking a day-long car trip, and learning geography.</span></span>

<span data-ttu-id="95c19-123">具有通常單一關聯式資料庫的單一應用程式有兩個重要優點：[ACID 交易](https://en.wikipedia.org/wiki/ACID)和 SQL 語言，兩者都可在與您應用程式相關的的所有資料表和資料之間使用。</span><span class="sxs-lookup"><span data-stu-id="95c19-123">A monolithic application with typically a single relational database has two important benefits: [ACID transactions](https://en.wikipedia.org/wiki/ACID) and the SQL language, both working across all the tables and data related to your application.</span></span> <span data-ttu-id="95c19-124">這種方法可用來輕鬆地撰寫查詢，以結合來自多個資料表的資料。</span><span class="sxs-lookup"><span data-stu-id="95c19-124">This approach provides a way to easily write a query that combines data from multiple tables.</span></span>

<span data-ttu-id="95c19-125">不過，當您移至微服務架構時，資料存取就會變得更複雜。</span><span class="sxs-lookup"><span data-stu-id="95c19-125">However, data access becomes much more complicated when you move to a microservices architecture.</span></span> <span data-ttu-id="95c19-126">即使在微服務或系結內容中使用 ACID 交易時，請務必考慮每個微服務所擁有的資料都是該微服務的私用，而且只能透過其 API 端點同步存取（REST、gRPC、SOAP 等等），或透過訊息（AMQP 或類似）以非同步方式進行。</span><span class="sxs-lookup"><span data-stu-id="95c19-126">Even when using ACID transactions within a microservice or Bounded Context, it is crucial to consider that the data owned by each microservice is private to that microservice and should only be accessed either synchronously through its API endpoints(REST, gRPC, SOAP, etc) or asynchronously via messaging(AMQP or similar).</span></span>

<span data-ttu-id="95c19-127">封裝資料可確保微服務鬆散偶合，而且彼此獨立地持續改進。</span><span class="sxs-lookup"><span data-stu-id="95c19-127">Encapsulating the data ensures that the microservices are loosely coupled and can evolve independently of one another.</span></span> <span data-ttu-id="95c19-128">如果多個服務存取相同的資料，結構描述更新便需要所有服的協調性更新。</span><span class="sxs-lookup"><span data-stu-id="95c19-128">If multiple services were accessing the same data, schema updates would require coordinated updates to all the services.</span></span> <span data-ttu-id="95c19-129">這會破壞微服務生命週期自主。</span><span class="sxs-lookup"><span data-stu-id="95c19-129">This would break the microservice lifecycle autonomy.</span></span> <span data-ttu-id="95c19-130">但分散式資料結構表示您無法讓單一 ACID 交易跨越微服務。</span><span class="sxs-lookup"><span data-stu-id="95c19-130">But distributed data structures mean that you can't make a single ACID transaction across microservices.</span></span> <span data-ttu-id="95c19-131">這又表示當商務程序跨越多個微服務時，您必須使用最終一致性。</span><span class="sxs-lookup"><span data-stu-id="95c19-131">This in turn means you must use eventual consistency when a business process spans multiple microservices.</span></span> <span data-ttu-id="95c19-132">這要實作時比簡單的 SQL 聯結難上許多，因為您無法建立完整性條件約束，或在不同的資料庫之間使用分散式交易，我們將在稍後說明。</span><span class="sxs-lookup"><span data-stu-id="95c19-132">This is much harder to implement than simple SQL joins, because you can't create integrity constraints or use distributed transactions between separate databases, as we'll explain later on.</span></span> <span data-ttu-id="95c19-133">同樣地，許多其他關聯式資料庫功能無法跨多個微服務使用。</span><span class="sxs-lookup"><span data-stu-id="95c19-133">Similarly, many other relational database features aren't available across multiple microservices.</span></span>

<span data-ttu-id="95c19-134">更進一步地說，不同的微服務通常使用不同「種類」的資料庫。</span><span class="sxs-lookup"><span data-stu-id="95c19-134">Going even further, different microservices often use different *kinds* of databases.</span></span> <span data-ttu-id="95c19-135">現代應用程式會存放和處理多種種類的資料，且關聯式資料庫不一定是最佳選擇。</span><span class="sxs-lookup"><span data-stu-id="95c19-135">Modern applications store and process diverse kinds of data, and a relational database isn't always the best choice.</span></span> <span data-ttu-id="95c19-136">對於某些使用案例，例如 Azure CosmosDB 或 MongoDB 的 NoSQL 資料庫，可能會有更方便的資料模型，並提供較 SQL Server 或 Azure SQL Database 等 SQL 資料庫更佳的效能和延展性。</span><span class="sxs-lookup"><span data-stu-id="95c19-136">For some use cases, a NoSQL database such as Azure CosmosDB or MongoDB might have a more convenient data model and offer better performance and scalability than a SQL database like SQL Server or Azure SQL Database.</span></span> <span data-ttu-id="95c19-137">在其他情況下，關聯式資料庫仍是最好的方法。</span><span class="sxs-lookup"><span data-stu-id="95c19-137">In other cases, a relational database is still the best approach.</span></span> <span data-ttu-id="95c19-138">因此，以微服務為基礎的應用程式通常會使用 SQL 和 NoSQL 資料庫的混合，這有時稱為[混合持續性 (Polyglot Persistence)](https://martinfowler.com/bliki/PolyglotPersistence.html) 方法。</span><span class="sxs-lookup"><span data-stu-id="95c19-138">Therefore, microservices-based applications often use a mixture of SQL and NoSQL databases, which is sometimes called the [polyglot persistence](https://martinfowler.com/bliki/PolyglotPersistence.html) approach.</span></span>

<span data-ttu-id="95c19-139">資料儲存體的資料分割、混合持續架構有許多優點。</span><span class="sxs-lookup"><span data-stu-id="95c19-139">A partitioned, polyglot-persistent architecture for data storage has many benefits.</span></span> <span data-ttu-id="95c19-140">這些包括鬆散偶合的服務和更佳的效能、延展性、成本與管理能力。</span><span class="sxs-lookup"><span data-stu-id="95c19-140">These include loosely coupled services and better performance, scalability, costs, and manageability.</span></span> <span data-ttu-id="95c19-141">不過，它可能導致某些分散式資料管理的挑戰，如本章稍後的[識別領域模型界限](identify-microservice-domain-model-boundaries.md)中所說明。</span><span class="sxs-lookup"><span data-stu-id="95c19-141">However, it can introduce some distributed data management challenges, as explained in "[Identifying domain-model boundaries](identify-microservice-domain-model-boundaries.md)" later in this chapter.</span></span>

## <a name="the-relationship-between-microservices-and-the-bounded-context-pattern"></a><span data-ttu-id="95c19-142">微服務和繫結內容模式之間的關聯性</span><span class="sxs-lookup"><span data-stu-id="95c19-142">The relationship between microservices and the Bounded Context pattern</span></span>

<span data-ttu-id="95c19-143">微服務概念衍生自[網域導向設計 (DDD)](https://martinfowler.com/bliki/BoundedContext.html) 中的[繫結內容 (BC) 模式](https://en.wikipedia.org/wiki/Domain-driven_design)。</span><span class="sxs-lookup"><span data-stu-id="95c19-143">The concept of microservice derives from the [Bounded Context (BC) pattern](https://martinfowler.com/bliki/BoundedContext.html) in [domain-driven design (DDD)](https://en.wikipedia.org/wiki/Domain-driven_design).</span></span> <span data-ttu-id="95c19-144">DDD 透過將大型模型分成多個 BC 並且明確指定其界限來處理大型模型。</span><span class="sxs-lookup"><span data-stu-id="95c19-144">DDD deals with large models by dividing them into multiple BCs and being explicit about their boundaries.</span></span> <span data-ttu-id="95c19-145">每個 BC 必須有自己的模型和資料庫。同樣地，每個微服務也擁有其相關資料。</span><span class="sxs-lookup"><span data-stu-id="95c19-145">Each BC must have its own model and database; likewise, each microservice owns its related data.</span></span> <span data-ttu-id="95c19-146">此外，每個 BC 通常都有它自己的[通用語言](https://martinfowler.com/bliki/UbiquitousLanguage.html)，協助軟體開發人員和網域專家之間的溝通。</span><span class="sxs-lookup"><span data-stu-id="95c19-146">In addition, each BC usually has its own [ubiquitous language](https://martinfowler.com/bliki/UbiquitousLanguage.html) to help communication between software developers and domain experts.</span></span>

<span data-ttu-id="95c19-147">通用語言中的詞彙 (主要為領域實體) 在不同繫結內容中可以有不同名稱，即使是不同領域實體都共用相同的識別 (亦即，用來從儲存體讀取實體的唯一識別碼)。</span><span class="sxs-lookup"><span data-stu-id="95c19-147">Those terms (mainly domain entities) in the ubiquitous language can have different names in different Bounded Contexts, even when different domain entities share the same identity (that is, the unique ID that's used to read the entity from storage).</span></span> <span data-ttu-id="95c19-148">比方說，在使用者設定檔繫結內容中，使用者網域實體可能會與排序之繫結內容中的買方網域實體共用身分識別。</span><span class="sxs-lookup"><span data-stu-id="95c19-148">For instance, in a user-profile Bounded Context, the User domain entity might share identity with the Buyer domain entity in the ordering Bounded Context.</span></span>

<span data-ttu-id="95c19-149">因此，微服務就像是繫結內容，但它也會指定其為一種分散式服務。</span><span class="sxs-lookup"><span data-stu-id="95c19-149">A microservice is therefore like a Bounded Context, but it also specifies that it's a distributed service.</span></span> <span data-ttu-id="95c19-150">它針對每個繫結內容建置為個別的處理序，且必須使用前文所述的分散式通訊協定，例如 HTTP/HTTPS、WebSockets 或 [AMQP](https://en.wikipedia.org/wiki/Advanced_Message_Queuing_Protocol)。</span><span class="sxs-lookup"><span data-stu-id="95c19-150">It's built as a separate process for each Bounded Context, and it must use the distributed protocols noted earlier, like HTTP/HTTPS, WebSockets, or [AMQP](https://en.wikipedia.org/wiki/Advanced_Message_Queuing_Protocol).</span></span> <span data-ttu-id="95c19-151">不過，繫結內容模式並未指定繫結內容是否為分散式服務，還是它只是單一部署應用程式中的邏輯界限 (例如泛型子系統)。</span><span class="sxs-lookup"><span data-stu-id="95c19-151">The Bounded Context pattern, however, doesn't specify whether the Bounded Context is a distributed service or if it's simply a logical boundary (such as a generic subsystem) within a monolithic-deployment application.</span></span>

<span data-ttu-id="95c19-152">值得強調的一點是，為每個繫結內容定義服務是個不錯的起點。</span><span class="sxs-lookup"><span data-stu-id="95c19-152">It's important to highlight that defining a service for each Bounded Context is a good place to start.</span></span> <span data-ttu-id="95c19-153">但您不必將設計限制於此。</span><span class="sxs-lookup"><span data-stu-id="95c19-153">But you don't have to constrain your design to it.</span></span> <span data-ttu-id="95c19-154">有時您必須設計由數個實體服務組成的繫結內容或商務微服務。</span><span class="sxs-lookup"><span data-stu-id="95c19-154">Sometimes you must design a Bounded Context or business microservice composed of several physical services.</span></span> <span data-ttu-id="95c19-155">但最後，這兩種模式 - 繫結內容和微服務 - 密切相關。</span><span class="sxs-lookup"><span data-stu-id="95c19-155">But ultimately, both patterns -Bounded Context and microservice- are closely related.</span></span>

<span data-ttu-id="95c19-156">DDD 藉由以分散式微服務的形式得到真實界限而從微服務獲益。</span><span class="sxs-lookup"><span data-stu-id="95c19-156">DDD benefits from microservices by getting real boundaries in the form of distributed microservices.</span></span> <span data-ttu-id="95c19-157">但如同不要在微服務之間共用模型，您可能也不想在繫結內容中共用模型。</span><span class="sxs-lookup"><span data-stu-id="95c19-157">But ideas like not sharing the model between microservices are what you also want in a Bounded Context.</span></span>
