# <a name="conclusion"></a>結論

以下要點是本指南中最重要的結論。

**使用無伺服器的好處。** 無伺服器解決方案提供成本節約的重要優勢，因為無伺服器是在按使用付費模型中實現的。 無伺服器功能可以獨立擴展、測試和部署應用程式的各個元件。 無伺服器非常適合實現微服務體系結構，並完全集成到 DevOps 管道中。

**代碼作為部署單元。** 無伺服器抽象硬體、作業系統和運行時，遠離應用程式。 無伺服器功能支援將代碼中的業務邏輯作為部署單元。

**觸發器和綁定。** 無伺服器可簡化與存儲、API 和其他雲資源的集成。 Azure 函數提供觸發器來執行代碼和綁定以與資源交互。

**微服務。** 微服務體系結構正在成為分散式、大型或複雜任務關鍵型應用程式的首選方法，這些應用程式基於多個獨立子系統，以自治服務的形式存在。 在基於微服務的體系結構中，應用程式構建為可獨立開發、測試、版本控制、部署和縮放的服務集合。 無伺服器是一種非常適合構建這些服務的體系結構。

**無伺服器平臺。** 無伺服器不僅僅是代碼的問題。 支援無伺服器體系結構的平臺包括無伺服器工作流和業務流程、無伺服器消息傳遞和事件服務以及無伺服器資料庫。

**無伺服器挑戰。** 無伺服器引入了與分散式應用程式開發相關的挑戰，例如碎片和獨立的資料模型、恢復能力、版本控制和服務發現。 無伺服器可能不適合長時間運行的進程或元件，這些流程或元件受益于更緊密的耦合。

**無伺服器作為工具，而不是工具箱。** 無伺服器不是應用程式體系結構的獨佔解決方案。 它是一種工具，可以作為混合應用程式的一部分加以利用，該應用程式可能包含傳統的層、單體後端和容器。 無伺服器可用於增強現有解決方案，不是應用程式開發全無方法。
