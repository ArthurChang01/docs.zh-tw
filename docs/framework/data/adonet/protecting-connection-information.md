## <a name="use-windows-authentication"></a><span data-ttu-id="372c6-108">使用 Windows 驗證。</span><span class="sxs-lookup"><span data-stu-id="372c6-108">Use Windows Authentication</span></span>  
 <span data-ttu-id="372c6-109">為了限制他人存取您的資料來源，您必須保護連線資訊，例如使用者 ID、密碼和資料來源名稱。</span><span class="sxs-lookup"><span data-stu-id="372c6-109">To help limit access to your data source, you must secure connection information such as user ID, password, and data source name.</span></span> <span data-ttu-id="372c6-110">為了避免公開使用者資訊，我們建議使用 Windows 驗證 (有時稱為*整合式安全性*) 可能的情況下。</span><span class="sxs-lookup"><span data-stu-id="372c6-110">In order to avoid exposing user information, we recommend using Windows authentication (sometimes referred to as *integrated security*) wherever possible.</span></span> <span data-ttu-id="372c6-111">Windows 驗證是藉由 `Integrated Security` 或 `Trusted_Connection` 關鍵字指定於連接字串中，可免除使用使用者 ID 和密碼的需要。</span><span class="sxs-lookup"><span data-stu-id="372c6-111">Windows authentication is specified in a connection string by using the `Integrated Security` or `Trusted_Connection` keywords, eliminating the need to use a user ID and password.</span></span> <span data-ttu-id="372c6-112">在使用 Windows 驗證時，使用者會由 Windows 進行驗證，而對伺服器和資料庫資源的存取權則是藉由授權給 Windows 使用者和群組來決定。</span><span class="sxs-lookup"><span data-stu-id="372c6-112">When using Windows authentication, users are authenticated by Windows, and access to server and database resources is determined by granting permissions to Windows users and groups.</span></span>  
  
 <span data-ttu-id="372c6-113">在無法使用 WIndows 驗證時必須特別小心，因為使用者認證會在連接字串中公開。</span><span class="sxs-lookup"><span data-stu-id="372c6-113">For situations where it is not possible to use Windows authentication, you must use extra care because user credentials are exposed in the connection string.</span></span> <span data-ttu-id="372c6-114">在 ASP.NET 應用程式中，可以將 Windows 帳戶設定為用於連接到資料庫和其他網路資源的固定識別 (Identity)。</span><span class="sxs-lookup"><span data-stu-id="372c6-114">In an ASP.NET application, you can configure a Windows account as a fixed identity that is used to connect to databases and other network resources.</span></span> <span data-ttu-id="372c6-115">啟用中的身分識別元素的模擬**web.config**檔案，並指定使用者名稱和密碼。</span><span class="sxs-lookup"><span data-stu-id="372c6-115">You enable impersonation in the identity element in the **web.config** file and specify a user name and password.</span></span>  
  
```xml

        userName="MyDomain\UserAccount"   
        password="*****" />  
```  
  
 <span data-ttu-id="372c6-116">固定識別帳戶應該是低權限的帳戶，僅為其授與資料庫中的必要權限。</span><span class="sxs-lookup"><span data-stu-id="372c6-116">The fixed identity account should be a low-privilege account that has been granted only necessary permissions in the database.</span></span> <span data-ttu-id="372c6-117">此外，您也應該將組態檔加密，讓使用者名稱和密碼不至於以純文字方式公開。</span><span class="sxs-lookup"><span data-stu-id="372c6-117">In addition, you should encrypt the configuration file so that the user name and password are not exposed in clear text.</span></span>

## <a name="avoid-injection-attacks-with-connection-string-builders"></a><span data-ttu-id="372c6-122">使用連接字串產生器來避免插入式攻擊</span><span class="sxs-lookup"><span data-stu-id="372c6-122">Avoid Injection Attacks with Connection String Builders</span></span>  
 <span data-ttu-id="372c6-123">當使用動態字串串連來根據使用者輸入建立連接字串時，就可能發生連接字串插入式攻擊。</span><span class="sxs-lookup"><span data-stu-id="372c6-123">A connection string injection attack can occur when dynamic string concatenation is used to build connection strings based on user input.</span></span> <span data-ttu-id="372c6-124">如果使用者輸入未經驗證且未逸出惡意的文字或字元，攻擊者就可能得以存取伺服器上的機密資料或其他資源。</span><span class="sxs-lookup"><span data-stu-id="372c6-124">If the user input is not validated and malicious text or characters not escaped, an attacker can potentially access sensitive data or other resources on the server.</span></span> <span data-ttu-id="372c6-125">為了解決此問題，ADO.NET 2.0 導入了新的連接字串產生器 (Builder) 類別 (Class)，可用於驗證連接字串語法，並確保沒有導入其他的參數。</span><span class="sxs-lookup"><span data-stu-id="372c6-125">To address this problem, ADO.NET 2.0 introduced new connection string builder classes to validate connection string syntax and ensure that additional parameters are not introduced.</span></span> <span data-ttu-id="372c6-126">如需詳細資訊，請參閱[連接字串產生器](../../../../docs/framework/data/adonet/connection-string-builders.md)。</span><span class="sxs-lookup"><span data-stu-id="372c6-126">For more information, see [Connection String Builders](../../../../docs/framework/data/adonet/connection-string-builders.md).</span></span>  
  
## <a name="use-persist-security-infofalse"></a><span data-ttu-id="372c6-127">使用 Persist Security Info=False</span><span class="sxs-lookup"><span data-stu-id="372c6-127">Use Persist Security Info=False</span></span>  
 <span data-ttu-id="372c6-128">`Persist Security Info` 的預設值為 False；建議您在所有連接字串中都使用此預設值。</span><span class="sxs-lookup"><span data-stu-id="372c6-128">The default value for `Persist Security Info` is false; we recommend using this default in all connection strings.</span></span> <span data-ttu-id="372c6-129">如果將 `Persist Security Info` 設定為 `true` 或 `yes`，則在開啟連接後，可透過連接取得安全機密資訊，包括使用者 ID 和密碼。</span><span class="sxs-lookup"><span data-stu-id="372c6-129">Setting `Persist Security Info` to `true` or `yes` allows security-sensitive information, including the user ID and password, to be obtained from a connection after it has been opened.</span></span> <span data-ttu-id="372c6-130">當 `Persist Security Info` 是設定為 `false` 或 `no` 時，安全性資訊會在用來開啟連接之後捨棄，以確保未受信任的來源無法存取安全機密資訊。</span><span class="sxs-lookup"><span data-stu-id="372c6-130">When `Persist Security Info` is set to `false` or `no`, security information is discarded after it is used to open the connection, ensuring that an untrusted source does not have access to security-sensitive information.</span></span>
