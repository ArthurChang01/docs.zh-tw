## <a name="stateless-versus-stateful-microservices"></a><span data-ttu-id="b6a00-149">無狀態微服務與具狀態微服務</span><span class="sxs-lookup"><span data-stu-id="b6a00-149">Stateless versus stateful microservices</span></span>

<span data-ttu-id="b6a00-150">如前所述，每項微服務 (邏輯繫結的內容) 都必須擁有其網域模型 (資料和邏輯)。</span><span class="sxs-lookup"><span data-stu-id="b6a00-150">As mentioned earlier, each microservice (logical Bounded Context) must own its domain model (data and logic).</span></span> <span data-ttu-id="b6a00-151">在無狀態微服務的案例中，資料庫會是外部的，採用如 SQL Server 的關聯式選項，或採用如 MongoDB 或 Azure Cosmos DB 的 NoSQL 選項。</span><span class="sxs-lookup"><span data-stu-id="b6a00-151">In the case of stateless microservices, the databases will be external, employing relational options like SQL Server, or NoSQL options like MongoDB or Azure Cosmos DB.</span></span>

<span data-ttu-id="b6a00-152">但是服務本身在 Service Fabric 中也可以具狀態，這表示資料位在微服務內。</span><span class="sxs-lookup"><span data-stu-id="b6a00-152">But the services themselves can also be stateful in Service Fabric, which means that the data resides within the microservice.</span></span> <span data-ttu-id="b6a00-153">這項資料可能不只存在於相同的伺服器，也存在微服務程序、記憶體內部，並保存在硬碟上，會複寫到其他節點。</span><span class="sxs-lookup"><span data-stu-id="b6a00-153">This data might exist not just on the same server, but within the microservice process, in memory and persisted on hard drives and replicated to other nodes.</span></span> <span data-ttu-id="b6a00-154">圖 4-30 顯示不同的方法。</span><span class="sxs-lookup"><span data-stu-id="b6a00-154">Figure 4-30 shows the different approaches.</span></span>

<span data-ttu-id="b6a00-155">**圖 4-30**。</span><span class="sxs-lookup"><span data-stu-id="b6a00-155">**Figure 4-30**.</span></span> <span data-ttu-id="b6a00-156">無狀態微服務與具狀態微服務</span><span class="sxs-lookup"><span data-stu-id="b6a00-156">Stateless versus stateful microservices</span></span>

<span data-ttu-id="b6a00-157">無狀態方法完全有效，而且比具狀態微服務更容易實作，因為方法類似於傳統和已知的模式。</span><span class="sxs-lookup"><span data-stu-id="b6a00-157">A stateless approach is perfectly valid and is easier to implement than stateful microservices, since the approach is similar to traditional and well-known patterns.</span></span> <span data-ttu-id="b6a00-158">但無狀態微服務會強制程序及資料來源之間的延遲。</span><span class="sxs-lookup"><span data-stu-id="b6a00-158">But stateless microservices impose latency between the process and data sources.</span></span> <span data-ttu-id="b6a00-159">當您嘗試使用額外的快取和佇列改善效能時，它們也牽涉到更多移動的片段。</span><span class="sxs-lookup"><span data-stu-id="b6a00-159">They also involve more moving pieces when you are trying to improve performance with additional cache and queues.</span></span> <span data-ttu-id="b6a00-160">您最後會得到有太多層的複雜架構。</span><span class="sxs-lookup"><span data-stu-id="b6a00-160">The result is that you can end up with complex architectures that have too many tiers.</span></span>

<span data-ttu-id="b6a00-161">相反地，[具狀態微服務](https://docs.microsoft.com/azure/service-fabric/service-fabric-reliable-services-introduction#when-to-use-reliable-services-apis)在進階案例中勝出，因為網域邏輯和資料之間沒有任何延遲。</span><span class="sxs-lookup"><span data-stu-id="b6a00-161">In contrast, [stateful microservices](https://docs.microsoft.com/azure/service-fabric/service-fabric-reliable-services-introduction#when-to-use-reliable-services-apis) can excel in advanced scenarios, because there is no latency between the domain logic and data.</span></span> <span data-ttu-id="b6a00-162">大量的資料處理、競先搶回終點、資料庫即服務，以及其他低延遲狀況皆得益於具狀態服務，它讓本機狀態可供更快速存取。</span><span class="sxs-lookup"><span data-stu-id="b6a00-162">Heavy data processing, gaming back ends, databases as a service, and other low-latency scenarios all benefit from stateful services, which enable local state for faster access.</span></span>
