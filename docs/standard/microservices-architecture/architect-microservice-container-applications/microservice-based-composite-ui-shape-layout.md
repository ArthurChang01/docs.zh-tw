# <a name="creating-composite-ui-based-on-microservices-including-visual-ui-shape-and-layout-generated-by-multiple-microservices"></a><span data-ttu-id="40bfa-103">根據微服務建立複合 UI，包括多個微服務所產生的視覺 UI 形狀和配置</span><span class="sxs-lookup"><span data-stu-id="40bfa-103">Creating composite UI based on microservices, including visual UI shape and layout generated by multiple microservices</span></span>

<span data-ttu-id="40bfa-104">微服務架構一開始通常是處理資料和邏輯的伺服器端。</span><span class="sxs-lookup"><span data-stu-id="40bfa-104">Microservices architecture often starts with the server side handling data and logic.</span></span> <span data-ttu-id="40bfa-105">不過，更進階的方法是同時根據微服務來設計應用程式 UI。</span><span class="sxs-lookup"><span data-stu-id="40bfa-105">However, a more advanced approach is to design your application UI based on microservices as well.</span></span> <span data-ttu-id="40bfa-106">這表示具有微服務所產生的複合 UI，而不是在伺服器上擁有多個微服務，並只由一個整合型用戶端應用程式來取用這些微服務。</span><span class="sxs-lookup"><span data-stu-id="40bfa-106">That means having a composite UI produced by the microservices, instead of having microservices on the server and just a monolithic client app consuming the microservices.</span></span> <span data-ttu-id="40bfa-107">透過此方法，您建置的微服務就會同時具備邏輯和視覺表示。</span><span class="sxs-lookup"><span data-stu-id="40bfa-107">With this approach, the microservices you build can be complete with both logic and visual representation.</span></span>

<span data-ttu-id="40bfa-108">圖 4-20 顯示更簡單的方法，該方法只會從整合型用戶端應用程式取用微服務。</span><span class="sxs-lookup"><span data-stu-id="40bfa-108">Figure 4-20 shows the simpler approach of just consuming microservices from a monolithic client application.</span></span> <span data-ttu-id="40bfa-109">當然，您在產生 HTML 與 JavaScript 之間可能會有 ASP.NET MVC 服務。</span><span class="sxs-lookup"><span data-stu-id="40bfa-109">Of course, you could have an ASP.NET MVC service in between producing the HTML and JavaScript.</span></span> <span data-ttu-id="40bfa-110">下圖已經過簡化，顯示您有取用微服務的單一 (整合型) 用戶端 UI，只著重於邏輯和資料，而不是 UI 形狀 (HTML 和 JavaScript)。</span><span class="sxs-lookup"><span data-stu-id="40bfa-110">The figure is a simplification that highlights that you have a single (monolithic) client UI consuming the microservices, which just focus on logic and data and not on the UI shape (HTML and JavaScript).</span></span>

<span data-ttu-id="40bfa-111">**圖 4-20**：</span><span class="sxs-lookup"><span data-stu-id="40bfa-111">**Figure 4-20**.</span></span> <span data-ttu-id="40bfa-112">取用後端微服務的整合型 UI 應用程式</span><span class="sxs-lookup"><span data-stu-id="40bfa-112">A monolithic UI application consuming back-end microservices</span></span>

<span data-ttu-id="40bfa-113">相較之下，複合 UI 是由微服務本身精確地產生和撰寫而成。</span><span class="sxs-lookup"><span data-stu-id="40bfa-113">In contrast, a composite UI is precisely generated and composed by the microservices themselves.</span></span> <span data-ttu-id="40bfa-114">部分微服務會支援特定 UI 區域的視覺形狀。</span><span class="sxs-lookup"><span data-stu-id="40bfa-114">Some of the microservices drive the visual shape of specific areas of the UI.</span></span> <span data-ttu-id="40bfa-115">主要差異在於您有以範本為基礎的用戶端 UI 元件 (例如 TS 類別)，而且這些範本的資料成形 UI ViewModel 來自每個微服務。</span><span class="sxs-lookup"><span data-stu-id="40bfa-115">The key difference is that you have client UI components (TS classes, for example) based on templates, and the data-shaping-UI ViewModel for those templates comes from each microservice.</span></span>

<span data-ttu-id="40bfa-116">用戶端應用程式啟動時，每個用戶端 UI 元件 (例如 TypeScript 類別) 會向基礎結構微服務註冊其本身，以便能夠針對指定案例提供 ViewModel。</span><span class="sxs-lookup"><span data-stu-id="40bfa-116">At client application start-up time, each of the client UI components (TypeScript classes, for example) registers itself with an infrastructure microservice capable of providing ViewModels for a given scenario.</span></span> <span data-ttu-id="40bfa-117">如果微服務變更形狀，UI 也會變更。</span><span class="sxs-lookup"><span data-stu-id="40bfa-117">If the microservice changes the shape, the UI changes also.</span></span>

<span data-ttu-id="40bfa-118">圖 4-21 顯示此複合 UI 方法的版本。</span><span class="sxs-lookup"><span data-stu-id="40bfa-118">Figure 4-21 shows a version of this composite UI approach.</span></span> <span data-ttu-id="40bfa-119">這已經過簡化，因為您可能會有根據不同技術彙總更細緻部分的其他微服務，這取決於您建置的是傳統 Web 方法 (ASP.NET MVC) 或 SPA (單頁應用程式)。</span><span class="sxs-lookup"><span data-stu-id="40bfa-119">This is simplified, because you might have other microservices that are aggregating granular parts based on different techniques—it depends on whether you are building a traditional web approach (ASP.NET MVC) or an SPA (Single Page Application).</span></span>

<span data-ttu-id="40bfa-120">**圖 4-21**：</span><span class="sxs-lookup"><span data-stu-id="40bfa-120">**Figure 4-21**.</span></span> <span data-ttu-id="40bfa-121">由後端微服務成形的複合 UI 應用程式範例</span><span class="sxs-lookup"><span data-stu-id="40bfa-121">Example of a composite UI application shaped by back-end microservices</span></span>

<span data-ttu-id="40bfa-122">每個 UI 組合微服務會類似於小型 API 閘道。</span><span class="sxs-lookup"><span data-stu-id="40bfa-122">Each of those UI composition microservices would be similar to a small API Gateway.</span></span> <span data-ttu-id="40bfa-123">但在本例中，每個微服務會負責一個很小的 UI 區域。</span><span class="sxs-lookup"><span data-stu-id="40bfa-123">But in this case each is responsible for a small UI area.</span></span>

<span data-ttu-id="40bfa-124">根據您所使用的 UI 技術，微服務導向的複合 UI 方法可能挑戰性很高，也可能很低。</span><span class="sxs-lookup"><span data-stu-id="40bfa-124">A composite UI approach that is driven by microservices can be more challenging or less so, depending on what UI technologies you are using.</span></span> <span data-ttu-id="40bfa-125">例如，您不會使用與用來建置 SPA 或原生行動應用程式相同的技術來建置傳統 Web 應用程式 (例如開發 Xamarin 應用程式時，此方法的挑戰性可能更高)。</span><span class="sxs-lookup"><span data-stu-id="40bfa-125">For instance, you will not use the same techniques for building a traditional web application that you use for building an SPA or for native mobile app (as when developing Xamarin apps, which can be more challenging for this approach).</span></span>

<span data-ttu-id="40bfa-126">[eShopOnContainers](https://aka.ms/MicroservicesArchitecture) 範例應用程式使用整合型 UI 方法的原因有很多。</span><span class="sxs-lookup"><span data-stu-id="40bfa-126">The [eShopOnContainers](https://aka.ms/MicroservicesArchitecture) sample application uses the monolithic UI approach for multiple reasons.</span></span> <span data-ttu-id="40bfa-127">首先，它是微服務和容器的前導。</span><span class="sxs-lookup"><span data-stu-id="40bfa-127">First, it is an introduction to microservices and containers.</span></span> <span data-ttu-id="40bfa-128">複合 UI 更進階，但在設計及開發 UI 時也需要更高的複雜度。</span><span class="sxs-lookup"><span data-stu-id="40bfa-128">A composite UI is more advanced but also requires further complexity when designing and developing the UI.</span></span> <span data-ttu-id="40bfa-129">其次，eShopOnContainers 也提供以 Xamarin 為基礎的原生行動應用程式，因此在用戶端 \# 側會更複雜。</span><span class="sxs-lookup"><span data-stu-id="40bfa-129">Second, eShopOnContainers also provides a native mobile app based on Xamarin, which would make it more complex on the client C\# side.</span></span>
