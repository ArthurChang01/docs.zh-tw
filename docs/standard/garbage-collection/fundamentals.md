### <a name="memory-allocation"></a><span data-ttu-id="4ec36-143">記憶體配置</span><span class="sxs-lookup"><span data-stu-id="4ec36-143">Memory allocation</span></span>

<span data-ttu-id="4ec36-144">當您初始化新的處理序 (Process) 時，Runtime 會保留一塊連續的位址空間區域，供處理序使用。</span><span class="sxs-lookup"><span data-stu-id="4ec36-144">When you initialize a new process, the runtime reserves a contiguous region of address space for the process.</span></span> <span data-ttu-id="4ec36-145">這塊保留的位址空間稱為 Managed 堆積 (Heap)。</span><span class="sxs-lookup"><span data-stu-id="4ec36-145">This reserved address space is called the managed heap.</span></span> <span data-ttu-id="4ec36-146">Managed 堆積會保留即將配置給堆積中下一個物件的位址指標。</span><span class="sxs-lookup"><span data-stu-id="4ec36-146">The managed heap maintains a pointer to the address where the next object in the heap will be allocated.</span></span> <span data-ttu-id="4ec36-147">剛開始會將這個指標設定為 Managed 堆積的基底位址 (Base Address)。</span><span class="sxs-lookup"><span data-stu-id="4ec36-147">Initially, this pointer is set to the managed heap's base address.</span></span> <span data-ttu-id="4ec36-148">所有參考類型都是在 Managed 堆積上進行配置。</span><span class="sxs-lookup"><span data-stu-id="4ec36-148">All reference types are allocated on the managed heap.</span></span> <span data-ttu-id="4ec36-149">當應用程式建立第一個參考型別時，會為該型別配置 Managed 堆積基底位址的記憶體。</span><span class="sxs-lookup"><span data-stu-id="4ec36-149">When an application creates the first reference type, memory is allocated for the type at the base address of the managed heap.</span></span> <span data-ttu-id="4ec36-150">當應用程式建立第二個物件時，記憶體回收行程會為它配置緊接在第一個物件後面位址空間的記憶體。</span><span class="sxs-lookup"><span data-stu-id="4ec36-150">When the application creates the next object, the garbage collector allocates memory for it in the address space immediately following the first object.</span></span> <span data-ttu-id="4ec36-151">只要有可供使用的位址空間，記憶體回收行程就會繼續用這種方式為新的物件配置空間。</span><span class="sxs-lookup"><span data-stu-id="4ec36-151">As long as address space is available, the garbage collector continues to allocate space for new objects in this manner.</span></span>

<span data-ttu-id="4ec36-152">從 Managed 堆積中配置記憶體要比 Unmanaged 記憶體配置快。</span><span class="sxs-lookup"><span data-stu-id="4ec36-152">Allocating memory from the managed heap is faster than unmanaged memory allocation.</span></span> <span data-ttu-id="4ec36-153">由於運行時通過向指標添加值來分配物件的記憶體,因此它幾乎與從堆疊中分配記憶體的速度一樣快。</span><span class="sxs-lookup"><span data-stu-id="4ec36-153">Because the runtime allocates memory for an object by adding a value to a pointer, it's almost as fast as allocating memory from the stack.</span></span> <span data-ttu-id="4ec36-154">此外,由於連續分配的新物件連續存儲在託管堆中,因此應用程式可以快速訪問物件。</span><span class="sxs-lookup"><span data-stu-id="4ec36-154">In addition, because new objects that are allocated consecutively are stored contiguously in the managed heap, an application can access the objects quickly.</span></span>

### <a name="memory-release"></a><span data-ttu-id="4ec36-155">記憶體釋放</span><span class="sxs-lookup"><span data-stu-id="4ec36-155">Memory release</span></span>

<span data-ttu-id="4ec36-156">記憶體回收行程的最佳化引擎會根據所做的配置，決定執行回收的最佳時機。</span><span class="sxs-lookup"><span data-stu-id="4ec36-156">The garbage collector's optimizing engine determines the best time to perform a collection based on the allocations being made.</span></span> <span data-ttu-id="4ec36-157">當記憶體回收行程執行回收時，會將應用程式已經不再使用的物件記憶體釋放出來。</span><span class="sxs-lookup"><span data-stu-id="4ec36-157">When the garbage collector performs a collection, it releases the memory for objects that are no longer being used by the application.</span></span> <span data-ttu-id="4ec36-158">它通過檢查應用程式的*根*來確定不再使用哪些物件。</span><span class="sxs-lookup"><span data-stu-id="4ec36-158">It determines which objects are no longer being used by examining the application's *roots*.</span></span> <span data-ttu-id="4ec36-159">應用程式的根目錄包括執行緒堆疊上的靜態欄位、區域變數和參數，以及 CPU 暫存器。</span><span class="sxs-lookup"><span data-stu-id="4ec36-159">An application's roots include static fields, local variables and parameters on a thread's stack, and CPU registers.</span></span> <span data-ttu-id="4ec36-160">每一個根目錄都會參考 Managed 堆積上的物件，要不然就是設定為 Null。</span><span class="sxs-lookup"><span data-stu-id="4ec36-160">Each root either refers to an object on the managed heap or is set to null.</span></span> <span data-ttu-id="4ec36-161">記憶體回收行程可以存取 Just-in-Time (JIT) 編譯器和執行階段所保留的使用中根目錄清單。</span><span class="sxs-lookup"><span data-stu-id="4ec36-161">The garbage collector has access to the list of active roots that the just-in-time (JIT) compiler and the runtime maintain.</span></span> <span data-ttu-id="4ec36-162">使用此清單,垃圾回收器將創建一個圖形,其中包含從根可接觸到的所有物件。</span><span class="sxs-lookup"><span data-stu-id="4ec36-162">Using this list, the garbage collector creates a graph that contains all the objects that are reachable from the roots.</span></span>

<span data-ttu-id="4ec36-163">不在圖形中的物件就無法從應用程式的根目錄取得。</span><span class="sxs-lookup"><span data-stu-id="4ec36-163">Objects that are not in the graph are unreachable from the application's roots.</span></span> <span data-ttu-id="4ec36-164">垃圾回收器考慮無法訪問的物件垃圾,並釋放為其分配的記憶體。</span><span class="sxs-lookup"><span data-stu-id="4ec36-164">The garbage collector considers unreachable objects garbage and releases the memory allocated for them.</span></span> <span data-ttu-id="4ec36-165">在回收期間，記憶體回收行程會檢查 Managed 堆積，尋找無法取得的物件所佔用的位址空間區塊。</span><span class="sxs-lookup"><span data-stu-id="4ec36-165">During a collection, the garbage collector examines the managed heap, looking for the blocks of address space occupied by unreachable objects.</span></span> <span data-ttu-id="4ec36-166">每找到一個無法取得的物件時，它便會使用記憶體複製功能，壓縮記憶體中可取得的物件，然後釋放出為無法取得的物件所配置的位址空間區塊。</span><span class="sxs-lookup"><span data-stu-id="4ec36-166">As it discovers each unreachable object, it uses a memory-copying function to compact the reachable objects in memory, freeing up the blocks of address spaces allocated to unreachable objects.</span></span> <span data-ttu-id="4ec36-167">一旦壓縮完可取得物件的記憶體之後，記憶體回收行程會進行必要的指標更正，使應用程式的根目錄指向新位置中的物件。</span><span class="sxs-lookup"><span data-stu-id="4ec36-167">Once the memory for the reachable objects has been compacted, the garbage collector makes the necessary pointer corrections so that the application's roots point to the objects in their new locations.</span></span> <span data-ttu-id="4ec36-168">它也會將 Managed 堆積的指標放在最後取得物件的後面。</span><span class="sxs-lookup"><span data-stu-id="4ec36-168">It also positions the managed heap's pointer after the last reachable object.</span></span>

<span data-ttu-id="4ec36-169">僅當集合發現大量無法訪問的物件時,才會壓縮記憶體。</span><span class="sxs-lookup"><span data-stu-id="4ec36-169">Memory is compacted only if a collection discovers a significant number of unreachable objects.</span></span> <span data-ttu-id="4ec36-170">如果 Managed 堆積中的所有物件在回收之後都存留下來，就不需要壓縮記憶體。</span><span class="sxs-lookup"><span data-stu-id="4ec36-170">If all the objects in the managed heap survive a collection, then there is no need for memory compaction.</span></span>

<span data-ttu-id="4ec36-171">為了改善效能，Runtime 會為大型物件配置不同堆積中的記憶體。</span><span class="sxs-lookup"><span data-stu-id="4ec36-171">To improve performance, the runtime allocates memory for large objects in a separate heap.</span></span> <span data-ttu-id="4ec36-172">記憶體回收行程會自動釋放大型物件的記憶體。</span><span class="sxs-lookup"><span data-stu-id="4ec36-172">The garbage collector automatically releases the memory for large objects.</span></span> <span data-ttu-id="4ec36-173">但是,為了避免在記憶體中移動大型物件,通常不壓縮此記憶體。</span><span class="sxs-lookup"><span data-stu-id="4ec36-173">However, to avoid moving large objects in memory, this memory is usually not compacted.</span></span>

## <a name="conditions-for-a-garbage-collection"></a><span data-ttu-id="4ec36-174">記憶體回收的條件</span><span class="sxs-lookup"><span data-stu-id="4ec36-174">Conditions for a garbage collection</span></span>

## <a name="the-managed-heap"></a><span data-ttu-id="4ec36-183">Managed 堆積</span><span class="sxs-lookup"><span data-stu-id="4ec36-183">The managed heap</span></span>

## <a name="generations"></a><span data-ttu-id="4ec36-202">層代</span><span class="sxs-lookup"><span data-stu-id="4ec36-202">Generations</span></span>

<span data-ttu-id="4ec36-203">GC 演算法基於以下幾個注意事項:</span><span class="sxs-lookup"><span data-stu-id="4ec36-203">The GC algorithm is based on several considerations:</span></span>

<span data-ttu-id="4ec36-207">垃圾回收主要與回收短壽命物件一起發生。</span><span class="sxs-lookup"><span data-stu-id="4ec36-207">Garbage collection primarily occurs with the reclamation of short-lived objects.</span></span> <span data-ttu-id="4ec36-208">為了優化垃圾回收器的性能,託管堆分為三代,0、1 和 2,以便可以分別處理長壽命和短壽命物件。</span><span class="sxs-lookup"><span data-stu-id="4ec36-208">To optimize the performance of the garbage collector, the managed heap is divided into three generations, 0, 1, and 2, so it can handle long-lived and short-lived objects separately.</span></span> <span data-ttu-id="4ec36-209">垃圾回收器將新物件存儲在第 0 代中。</span><span class="sxs-lookup"><span data-stu-id="4ec36-209">The garbage collector stores new objects in generation 0.</span></span> <span data-ttu-id="4ec36-210">在應用程式存留期初期建立的物件則會升階並儲存在第 1 個和第 2 個層代。</span><span class="sxs-lookup"><span data-stu-id="4ec36-210">Objects created early in the application's lifetime that survive collections are promoted and stored in generations 1 and 2.</span></span> <span data-ttu-id="4ec36-211">由於壓縮託管堆的一部分比整個堆快,因此此方案允許垃圾回收器在特定生成中釋放記憶體,而不是在每次執行集合時釋放整個託管堆的記憶體。</span><span class="sxs-lookup"><span data-stu-id="4ec36-211">Because it's faster to compact a portion of the managed heap than the entire heap, this scheme allows the garbage collector to release the memory in a specific generation rather than release the memory for the entire managed heap each time it performs a collection.</span></span>

- <span data-ttu-id="4ec36-212">**第 0 代**.</span><span class="sxs-lookup"><span data-stu-id="4ec36-212">**Generation 0**.</span></span> <span data-ttu-id="4ec36-213">這是最新的層代而且包含存留較短的物件。</span><span class="sxs-lookup"><span data-stu-id="4ec36-213">This is the youngest generation and contains short-lived objects.</span></span> <span data-ttu-id="4ec36-214">存留較短的物件範例是暫存變數。</span><span class="sxs-lookup"><span data-stu-id="4ec36-214">An example of a short-lived object is a temporary variable.</span></span> <span data-ttu-id="4ec36-215">記憶體回收最常在這個層代中進行。</span><span class="sxs-lookup"><span data-stu-id="4ec36-215">Garbage collection occurs most frequently in this generation.</span></span>

  <span data-ttu-id="4ec36-216">新分配的對象構成新一代的物件,並且隱式生成 0 集合。</span><span class="sxs-lookup"><span data-stu-id="4ec36-216">Newly allocated objects form a new generation of objects and are implicitly generation 0 collections.</span></span> <span data-ttu-id="4ec36-217">但是,如果它們是大型對象,它們將位於第 2 代集合中的大物件堆上。</span><span class="sxs-lookup"><span data-stu-id="4ec36-217">However, if they are large objects, they go on the large object heap in a generation 2 collection.</span></span>

- <span data-ttu-id="4ec36-222">**第 1 代**.</span><span class="sxs-lookup"><span data-stu-id="4ec36-222">**Generation 1**.</span></span> <span data-ttu-id="4ec36-223">這個層代包含存留較短的物件，而且當做存留較短物件與存留較長物件之間的緩衝區。</span><span class="sxs-lookup"><span data-stu-id="4ec36-223">This generation contains short-lived objects and serves as a buffer between short-lived objects and long-lived objects.</span></span>

- <span data-ttu-id="4ec36-229">**第 2 代**.</span><span class="sxs-lookup"><span data-stu-id="4ec36-229">**Generation 2**.</span></span> <span data-ttu-id="4ec36-230">這個層代包含存留較長的物件。</span><span class="sxs-lookup"><span data-stu-id="4ec36-230">This generation contains long-lived objects.</span></span> <span data-ttu-id="4ec36-231">長壽命物件的一個範例是伺服器應用程式中包含在進程持續時間內即時的靜態數據的物件。</span><span class="sxs-lookup"><span data-stu-id="4ec36-231">An example of a long-lived object is an object in a server application that contains static data that's live for the duration of the process.</span></span>

## <a name="what-happens-during-a-garbage-collection"></a><span data-ttu-id="4ec36-270">記憶體回收期間進行的作業</span><span class="sxs-lookup"><span data-stu-id="4ec36-270">What happens during a garbage collection</span></span>

<span data-ttu-id="4ec36-271">記憶體回收具有下列階段：</span><span class="sxs-lookup"><span data-stu-id="4ec36-271">A garbage collection has the following phases:</span></span>

- <span data-ttu-id="4ec36-272">標記階段：尋找和建立所有使用中物件的清單。</span><span class="sxs-lookup"><span data-stu-id="4ec36-272">A marking phase that finds and creates a list of all live objects.</span></span>

- <span data-ttu-id="4ec36-273">重新配置階段：更新即將壓縮之物件的參考。</span><span class="sxs-lookup"><span data-stu-id="4ec36-273">A relocating phase that updates the references to the objects that will be compacted.</span></span>

- <span data-ttu-id="4ec36-274">壓縮階段：回收無作用物件所佔據的空間並壓縮未被回收的物件。</span><span class="sxs-lookup"><span data-stu-id="4ec36-274">A compacting phase that reclaims the space occupied by the dead objects and compacts the surviving objects.</span></span> <span data-ttu-id="4ec36-275">壓縮階段會將記憶體回收中未被回收的物件移至區段的較舊端。</span><span class="sxs-lookup"><span data-stu-id="4ec36-275">The compacting phase moves objects that have survived a garbage collection toward the older end of the segment.</span></span>

  <span data-ttu-id="4ec36-276">因為層代 2 回收可能會佔據多個區段，所以提升至層代 2 的物件可能會移至較舊區段。</span><span class="sxs-lookup"><span data-stu-id="4ec36-276">Because generation 2 collections can occupy multiple segments, objects that are promoted into generation 2 can be moved into an older segment.</span></span> <span data-ttu-id="4ec36-277">層代 1 和層代 2 的未回收物件都可能會移至不同的區段，因為它們都會被提升至層代 2。</span><span class="sxs-lookup"><span data-stu-id="4ec36-277">Both generation 1 and generation 2 survivors can be moved to a different segment, because they are promoted to generation 2.</span></span>

  <span data-ttu-id="4ec36-278">通常,大型物件堆 (LOH) 不會壓縮,因為複製大型物件會造成性能損失。</span><span class="sxs-lookup"><span data-stu-id="4ec36-278">Ordinarily, the large object heap (LOH) is not compacted, because copying large objects imposes a performance penalty.</span></span> <span data-ttu-id="4ec36-279">但是,在 .NET Core 和 .NET 框架<xref:System.Runtime.GCSettings.LargeObjectHeapCompactionMode%2A?displayProperty=nameWithType>4.5.1 及更高版本中,可以使用 屬性按需壓縮大型物件堆。</span><span class="sxs-lookup"><span data-stu-id="4ec36-279">However, in .NET Core and in .NET Framework 4.5.1 and later, you can use the <xref:System.Runtime.GCSettings.LargeObjectHeapCompactionMode%2A?displayProperty=nameWithType> property to compact the large object heap on demand.</span></span> <span data-ttu-id="4ec36-280">此外,通過指定以下任一條件設置硬限制時,LOH 會自動壓縮:</span><span class="sxs-lookup"><span data-stu-id="4ec36-280">In addition, the LOH is automatically compacted when a hard limit is set by specifying either:</span></span>

  - <span data-ttu-id="4ec36-281">容器上的記憶體限制。</span><span class="sxs-lookup"><span data-stu-id="4ec36-281">A memory limit on a container.</span></span>
  - <span data-ttu-id="4ec36-282">[GCHeapHard 限制](../../core/run-time-config/garbage-collector.md#systemgcheaphardlimitcomplus_gcheaphardlimit)或[GCHeapHard 限制百分比](../../core/run-time-config/garbage-collector.md#systemgcheaphardlimitpercentcomplus_gcheaphardlimitpercent)運行時配置選項。</span><span class="sxs-lookup"><span data-stu-id="4ec36-282">The [GCHeapHardLimit](../../core/run-time-config/garbage-collector.md#systemgcheaphardlimitcomplus_gcheaphardlimit) or [GCHeapHardLimitPercent](../../core/run-time-config/garbage-collector.md#systemgcheaphardlimitpercentcomplus_gcheaphardlimitpercent) run-time configuration options.</span></span>

<span data-ttu-id="4ec36-283">記憶體回收行程會使用下列資訊來判斷物件是否使用中：</span><span class="sxs-lookup"><span data-stu-id="4ec36-283">The garbage collector uses the following information to determine whether objects are live:</span></span>

- <span data-ttu-id="4ec36-287">**垃圾資源回收桿 。**</span><span class="sxs-lookup"><span data-stu-id="4ec36-287">**Garbage collection handles**.</span></span> <span data-ttu-id="4ec36-288">會指向 Managed 物件，而且可由使用者程式碼或 Common Language Runtime 配置的控制代碼。</span><span class="sxs-lookup"><span data-stu-id="4ec36-288">Handles that point to managed objects and that can be allocated by user code or by the common language runtime.</span></span>
