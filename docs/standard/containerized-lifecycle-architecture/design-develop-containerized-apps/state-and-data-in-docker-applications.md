# <a name="state-and-data-in-docker-applications"></a><span data-ttu-id="bfbea-103">Docker 應用程式中的狀態和資料</span><span class="sxs-lookup"><span data-stu-id="bfbea-103">State and data in Docker applications</span></span>

<span data-ttu-id="bfbea-104">在大多數情況下，您可以將容器視為處理序執行個體。</span><span class="sxs-lookup"><span data-stu-id="bfbea-104">In most cases, you can think of a container as an instance of a process.</span></span> <span data-ttu-id="bfbea-105">處理序不會保留持續性狀態。</span><span class="sxs-lookup"><span data-stu-id="bfbea-105">A process does not maintain persistent state.</span></span> <span data-ttu-id="bfbea-106">雖然容器可以寫入其本機存放區，假設執行個體將會解決無限期地就像是假設記憶體中的單一位置會持久中。</span><span class="sxs-lookup"><span data-stu-id="bfbea-106">While a container can write to its local storage, assuming that an instance will be around indefinitely is like assuming that a single location in memory will be durable.</span></span> <span data-ttu-id="bfbea-107">容器映像，例如處理序應該假設多個執行個體，而且它們會最後終究會終止;如果它們由容器協調器管理，則應該假設，它們可能會取得一個節點或 VM 之間移動。</span><span class="sxs-lookup"><span data-stu-id="bfbea-107">Container images, like processes, should be assumed to have multiple instances and that they will eventually be killed; if they're managed with a container orchestrator, it should be assumed that they might get moved from one node or VM to another.</span></span>

<span data-ttu-id="bfbea-108">下列解決方案可用來管理 Docker 應用程式中的持續性資料：</span><span class="sxs-lookup"><span data-stu-id="bfbea-108">The following solutions are used to manage persistent data in Docker applications:</span></span>

<span data-ttu-id="bfbea-109">從 Docker 主機，透過 [Docker Volumes](https://docs.docker.com/engine/admin/volumes/) (Docker 磁碟區)：</span><span class="sxs-lookup"><span data-stu-id="bfbea-109">From the Docker host, as [Docker Volumes](https://docs.docker.com/engine/admin/volumes/):</span></span>

- <span data-ttu-id="bfbea-110">**磁碟區**會儲存在主機檔案系統中由 Docker 管理的一個區域內。</span><span class="sxs-lookup"><span data-stu-id="bfbea-110">**Volumes** are stored in an area of the host filesystem that's managed by Docker.</span></span>

- <span data-ttu-id="bfbea-111">**繫結掛接**可以對應至主機檔案系統中任何資料夾中，以便存取無法控制從 Docker 處理序，而且可能會造成安全性風險，容器可能存取機密的作業系統資料夾。</span><span class="sxs-lookup"><span data-stu-id="bfbea-111">**Bind mounts** can map to any folder in the host filesystem, so access can't be controlled from a Docker process and can pose a security risk as a container could access sensitive OS folders.</span></span>

- <span data-ttu-id="bfbea-112">**tmpfs 裝載**則與虛擬資料夾相似，只存在於主機的記憶體中，永遠不會寫入檔案系統。</span><span class="sxs-lookup"><span data-stu-id="bfbea-112">**tmpfs mounts** are like virtual folders that only exist in the host's memory and are never written to the filesystem.</span></span>

<span data-ttu-id="bfbea-113">從遠端存放：</span><span class="sxs-lookup"><span data-stu-id="bfbea-113">From remote storage:</span></span>

- <span data-ttu-id="bfbea-114">[Azure 儲存體](https://azure.microsoft.com/documentation/services/storage/)提供備有異地分散式儲存體，適用於容器提供很好的長期持續性解決方案。</span><span class="sxs-lookup"><span data-stu-id="bfbea-114">[Azure Storage](https://azure.microsoft.com/documentation/services/storage/) provides geo-distributable storage, providing a good long-term persistence solution for containers.</span></span>

- <span data-ttu-id="bfbea-115">遠端關聯式資料庫喜歡[Azure SQL Database](https://azure.microsoft.com/services/sql-database/)，例如 NoSQL 資料庫[Azure Cosmos DB](https://docs.microsoft.com/azure/cosmos-db/introduction)，或快取服務，例如[Redis](https://redis.io/)。</span><span class="sxs-lookup"><span data-stu-id="bfbea-115">Remote relational databases like [Azure SQL Database](https://azure.microsoft.com/services/sql-database/), NoSQL databases like [Azure Cosmos DB](https://docs.microsoft.com/azure/cosmos-db/introduction), or cache services like [Redis](https://redis.io/).</span></span>

<span data-ttu-id="bfbea-116">從 Docker 容器：</span><span class="sxs-lookup"><span data-stu-id="bfbea-116">From the Docker container:</span></span>

- <span data-ttu-id="bfbea-117">Docker 提供一項功能，稱為「重疊檔案系統」。</span><span class="sxs-lookup"><span data-stu-id="bfbea-117">Docker provides a feature named the *overlay file system*.</span></span> <span data-ttu-id="bfbea-118">這項功能會實作寫入時複製工作存放區更新容器的根檔案系統的資訊。</span><span class="sxs-lookup"><span data-stu-id="bfbea-118">This feature implements a copy-on-write task that stores updated information to the root file system of the container.</span></span> <span data-ttu-id="bfbea-119">該資訊"配置的最上層顯示 」 容器所依據的原始映像。</span><span class="sxs-lookup"><span data-stu-id="bfbea-119">That information "lays on top of" the original image on which the container is based.</span></span> <span data-ttu-id="bfbea-120">如果從系統中刪除容器，這些變更將會遺失。</span><span class="sxs-lookup"><span data-stu-id="bfbea-120">If the container is deleted from the system, those changes are lost.</span></span> <span data-ttu-id="bfbea-121">因此，雖然您可以儲存在其本機儲存體容器的狀態，設計這項功能為基礎的系統容器的設計，其預設值是無狀態的內部部署與會衝突。</span><span class="sxs-lookup"><span data-stu-id="bfbea-121">Therefore, while it's possible to save the state of a container within its local storage, designing a system based on this feature would conflict with the premise of container design, which by default is stateless.</span></span>

- <span data-ttu-id="bfbea-122">不過，Docker 磁碟區現在是慣用的方法來處理本機在 Docker 中的資料。</span><span class="sxs-lookup"><span data-stu-id="bfbea-122">However, Docker Volumes is now the preferred way to handle local data in Docker.</span></span> <span data-ttu-id="bfbea-123">如果您需要在容器中的儲存體的詳細資訊，請檢查[Docker 存放裝置驅動程式](https://docs.docker.com/engine/userguide/storagedriver/)並[映像、 容器和存放裝置驅動程式的相關](https://docs.docker.com/engine/userguide/storagedriver/imagesandcontainers/)。</span><span class="sxs-lookup"><span data-stu-id="bfbea-123">If you need more information about storage in containers, check on [Docker storage drivers](https://docs.docker.com/engine/userguide/storagedriver/) and [About images, containers, and storage drivers](https://docs.docker.com/engine/userguide/storagedriver/imagesandcontainers/).</span></span>

<span data-ttu-id="bfbea-124">以下提供有關這些選項的其他詳細資料。</span><span class="sxs-lookup"><span data-stu-id="bfbea-124">The following provides additional detail about these options.</span></span>

<span data-ttu-id="bfbea-125">**磁碟區**是從主機 OS 對應至容器內目錄的目錄。</span><span class="sxs-lookup"><span data-stu-id="bfbea-125">**Volumes** are directories mapped from the host OS to directories in containers.</span></span> <span data-ttu-id="bfbea-126">當容器中的程式碼具有目錄的存取權時，該存取權實際上就是主機 OS 上的目錄存取權。</span><span class="sxs-lookup"><span data-stu-id="bfbea-126">When code in the container has access to the directory, that access is actually to a directory on the host OS.</span></span> <span data-ttu-id="bfbea-127">此目錄不會和容器本身的存留期繫結，且目錄會由 Docker 管理，且和主機電腦的核心功能隔離。</span><span class="sxs-lookup"><span data-stu-id="bfbea-127">This directory is not tied to the lifetime of the container itself, and the directory is managed by Docker and isolated from the core functionality of the host machine.</span></span> <span data-ttu-id="bfbea-128">因此，資料磁碟區設計成可無視於容器的存留期來保存資料。</span><span class="sxs-lookup"><span data-stu-id="bfbea-128">Thus, data volumes are designed to persist data independently of the life of the container.</span></span> <span data-ttu-id="bfbea-129">如果您從 Docker 主機中刪除容器或映像，則不會刪除保存在資料磁碟區中的資料。</span><span class="sxs-lookup"><span data-stu-id="bfbea-129">If you delete a container or an image from the Docker host, the data persisted in the data volume is not deleted.</span></span>

<span data-ttu-id="bfbea-130">磁碟區可以具名，也可以是匿名 (預設)。</span><span class="sxs-lookup"><span data-stu-id="bfbea-130">Volumes can be named or anonymous (the default).</span></span> <span data-ttu-id="bfbea-131">具名磁碟區是**資料磁碟區容器**的演進，可輕鬆的在容器間共用資料。</span><span class="sxs-lookup"><span data-stu-id="bfbea-131">Named volumes are the evolution of **Data Volume Containers** and make it easy to share data between containers.</span></span> <span data-ttu-id="bfbea-132">磁碟區也支援可讓您將資料儲存在其他選項之間的遠端主機上的磁碟區驅動程式。</span><span class="sxs-lookup"><span data-stu-id="bfbea-132">Volumes also support volume drivers that allow you to store data on remote hosts, among other options.</span></span>

<span data-ttu-id="bfbea-133">**繫結掛接**已提供了很長的時間，並允許在容器中的掛接點資料夾的對應。</span><span class="sxs-lookup"><span data-stu-id="bfbea-133">**Bind mounts** have been available for a long time and allow the mapping of any folder to a mount point in a container.</span></span> <span data-ttu-id="bfbea-134">繫結裝載的限制比磁碟區更多，且具有某些重要的安全性問題，因此磁碟區是建議的選項。</span><span class="sxs-lookup"><span data-stu-id="bfbea-134">Bind mounts have more limitations than volumes and some important security issues, so volumes are the recommended option.</span></span>

<span data-ttu-id="bfbea-135">**`tmpfs` 掛接**是 live 只在主機的記憶體中，永遠不會寫到檔案系統的虛擬資料夾。</span><span class="sxs-lookup"><span data-stu-id="bfbea-135">**`tmpfs` mounts** are virtual folders that live only in the host's memory and are never written to the filesystem.</span></span> <span data-ttu-id="bfbea-136">它們既快速又安全，但會使用記憶體，且僅適用於非持續性資料。</span><span class="sxs-lookup"><span data-stu-id="bfbea-136">They are fast and secure but use memory and are only meant for non-persistent data.</span></span>

<span data-ttu-id="bfbea-137">如圖 4-5 所示，一般 Docker 磁碟區可儲存在容器本身之外，但必須在主機伺服器或 VM 的實體界限內。</span><span class="sxs-lookup"><span data-stu-id="bfbea-137">As shown in Figure 4-5, regular Docker volumes can be stored outside of the containers themselves but within the physical boundaries of the host server or VM.</span></span> <span data-ttu-id="bfbea-138">不過，Docker 容器無法從一部主機伺服器或 VM 存取另一部主機伺服器或 VM 的磁碟區。</span><span class="sxs-lookup"><span data-stu-id="bfbea-138">However, Docker containers cannot access a volume from one host server or VM to another.</span></span> <span data-ttu-id="bfbea-139">換句話說，使用這些磁碟區，將無法管理在不同 Docker 主機上執行容器間所共用的資料，雖然仍可透過支援遠端主機的磁碟區驅動程式來達到此目的。</span><span class="sxs-lookup"><span data-stu-id="bfbea-139">In other words, with these volumes, it isn't possible to manage data shared between containers that run on different Docker hosts, although it could be achieved with a volume driver that supports remote hosts.</span></span>

<span data-ttu-id="bfbea-143">此外，當 Docker 容器是由協調器所管理時，容器可能會根據叢集所執行的最佳化，在主機之間「移動」。</span><span class="sxs-lookup"><span data-stu-id="bfbea-143">In addition, when Docker containers are managed by an orchestrator, containers might "move" between hosts, depending on the optimizations performed by the cluster.</span></span> <span data-ttu-id="bfbea-144">因此，不建議您使用資料磁碟區來儲存商務資料。</span><span class="sxs-lookup"><span data-stu-id="bfbea-144">Therefore, it isn't recommended that you use data volumes for business data.</span></span> <span data-ttu-id="bfbea-145">但它們的好方法來使用追蹤檔案、 時態性檔案，或類似，可將不會影響商務資料的一致性。</span><span class="sxs-lookup"><span data-stu-id="bfbea-145">But they are a good mechanism to work with trace files, temporal files, or similar, that will not impact business data consistency.</span></span>

<span data-ttu-id="bfbea-146">**遠端資料來源和快取**工具 (例如 Azure SQL Database、Azure Cosmos DB) 或遠端快取 (例如 Redis) 可用於容器化應用程式，就像是在沒有容器時用於開發一樣。</span><span class="sxs-lookup"><span data-stu-id="bfbea-146">**Remote data sources and cache** tools like Azure SQL Database, Azure Cosmos DB, or a remote cache like Redis can be used in containerized applications the same way they are used when developing without containers.</span></span> <span data-ttu-id="bfbea-147">此方法經過實證，可儲存商務應用程式資料。</span><span class="sxs-lookup"><span data-stu-id="bfbea-147">This is a proven way to store business application data.</span></span>

<span data-ttu-id="bfbea-148">**Azure 儲存體。**</span><span class="sxs-lookup"><span data-stu-id="bfbea-148">**Azure Storage.**</span></span> <span data-ttu-id="bfbea-149">商務資料通常需要放在外部資源或資料庫，例如 Azure 儲存體。</span><span class="sxs-lookup"><span data-stu-id="bfbea-149">Business data usually needs to be placed in external resources or databases, like Azure Storage.</span></span> <span data-ttu-id="bfbea-150">Azure 儲存體提供雲端中的下列服務：</span><span class="sxs-lookup"><span data-stu-id="bfbea-150">Azure Storage provides the following services in the cloud:</span></span>
